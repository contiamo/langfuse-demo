Observability · Tracing
Structured, per-request logs of every LLM call — prompt, response, tokens, latency, and intermediate steps — captured as your app runs.
The Problem
AI is non-deterministic: the same input can produce different outputs. Without a trace, you can't reproduce a failure, find where latency spiked, or know exactly what prompt was sent and what came back.
The Concept
A trace records one full request: prompt, model response, token counts, latency at each step, and any tool calls or retrieval steps. Traces send asynchronously — zero latency impact on your app.
How to use it
  • Decorate your Python functions with @observe() — it auto-captures prompts, completions, and token usage.
  • Tag traces with environment, user_id, and custom metadata for later filtering in the UI.
  • View the trace tree in Langfuse: expand any span to inspect the exact prompt text, model parameters, latency, and cost.
  • Use custom trace_id to correlate with your own distributed tracing systems.
Key Terms
trace span / observation generation @observe() async flush
Observability · Sessions
Group multiple traces from a single user conversation into one replayable session — debug multi-turn interactions end to end.
The Problem
Multi-turn conversations produce separate traces. Without sessions, you only see isolated fragments — you can't replay a conversation, evaluate it as a whole, or understand why turn 4 failed given turns 1–3.
The Concept
A session is a named group of traces linked by a shared session_id string. Langfuse renders them as a conversation replay in order. Any trace tagged with the same ID is automatically collected — no extra work per turn.
How to use it
  • Pass session_id via propagate_attributes() — all nested observations inherit it, including OpenAI and LangChain integrations.
  • The ID is any US-ASCII string under 200 chars — your existing chat/thread ID works perfectly.
  • Add session-level scores via SDK/API (e.g. from end-of-conversation user feedback).
  • Share a public session link for async review with teammates or non-technical stakeholders.
Key Terms
session_id session replay propagate_attributes() multi-turn
Observability · Users
Map every trace, cost figure, and feedback score to individual users via a user_id — then segment, compare, and deep-dive per person.
The Problem
Without user tracking, you see aggregate LLM usage but can't answer: which user is driving cost? Who gives negative feedback? How is a specific user's experience trending? Debugging a production complaint requires filtering all traces to that person.
The Concept
Set user_id (any stable string — email, UUID, username) on traces via propagate_attributes(). Langfuse auto-aggregates token cost, trace count, and scores per user. The Users view provides both a list for segmentation and a detail view for per-user deep dives.
How to use it
  • Set propagate_attributes(user_id="...") inside any @observe()-decorated function — propagates to all nested spans automatically.
  • Open the Users list in the UI to rank users by total token cost, trace volume, or quality score.
  • Click any user to see their full trace history — filter by time range or score to find anomalies.
  • Query per-user metrics programmatically via the Metrics API for custom dashboards.
Key Terms
user_id propagate_attributes() per-user cost Metrics API
Observability · Dashboards & Metrics
Aggregate traces into sliceable dashboards covering quality, cost, latency, and volume — sliced by user, model, prompt version, or tag.
The Problem
Raw traces tell you what happened on one request. They don't answer: is quality improving across prompt versions? Which model costs the most? Which feature drives the most traffic? You need aggregation.
The Concept
Langfuse Metrics derive structured analytics from traces: quality scores, cost and latency per user/session/model, and volume trends. All sliceable by trace name, userId, tags, and release strings. Export to PostHog or Mixpanel, or query the Metrics API directly.
How to use it
  • Add a name field to traces to differentiate features — this becomes the primary slice dimension in dashboards.
  • Add tags to group experiments, prompt variants, or deployment environments for filtered views.
  • Set release version strings on traces so you can compare metric trends across prompt or model changes.
  • Use the customizable Dashboard UI or the Metrics API to pull data into your own tooling.
Key Terms
trace name tags release / version Metrics API PostHog export
Observability · Monitors (Beta)
Threshold-based alerting on LLM metrics — fire Slack messages, webhooks, or GitHub Actions when cost spikes or quality drops.
The Problem
Cost spikes and quality regressions in LLM apps are usually discovered by users before engineers. Continuously checking dashboards manually doesn't scale. You need alerts that wake you up when something crosses a threshold.
The Concept
A Monitor evaluates a metric query on a rolling time window (1h, 1d, etc.) and compares the result against thresholds. Severity transitions (OK → WARNING → ALERT) each fire linked Automations — Slack channel, HTTP webhook (with HMAC signature), or GitHub Actions workflow_dispatch.
How to set up
  • Define the metric: choose data source (Observations, Scores), aggregation (avg, count, p95), and filters (model, tag, environment).
  • Set thresholds: comparison operator, warning level, alert level, and the evaluation window.
  • Create an Automation first (Slack, webhook, or GitHub), then link it to the monitor.
  • Save — monitor goes ACTIVE immediately. After 5 consecutive delivery failures, the trigger auto-disables.
Key Terms
Monitor Severity (OK/WARNING/ALERT) Automation Evaluation window

Prompt Management · Prompts
Store, version, and serve prompts centrally — your code fetches the live production version at runtime. No redeploy needed to update a prompt.
The Problem
Prompts hardcoded in source code require a deploy to change — slow and risky. When prompts are scattered across the codebase, there is no audit trail: who changed it, when, and what did it say before?
The Concept
Prompts are versioned in Langfuse with a name, type (text or chat), and {{variable}} template syntax. The production label marks the active version. The SDK fetches it at runtime and compiles variables locally — no extra network round-trip in your hot path.
How to use it
  • Create: langfuse.create_prompt(name="rag-answer", prompt="Answer {{question}}...", labels=["production"])
  • Fetch at runtime: prompt = langfuse.get_prompt("rag-answer") — returns the production-labeled version by default.
  • Compile: compiled = prompt.compile(question="Why is the sky blue?") — returns the final string ready to send.
  • Pin a version: get_prompt("name", version=3) — for reproducible experiments.
Key Terms
prompt version label: production compile() text vs chat type get_prompt()
Prompt Management · Playground
A browser-based workbench for testing and refining prompts against live LLM providers — without leaving Langfuse or writing a line of code.
The Problem
Iterating on prompts means switching between code editors, API clients, and observability tools — slow feedback loops. Debugging a bad generation means re-running code just to test a small tweak.
The Concept
Edit prompt text, set variables, adjust model params, and test tool-calling — all in the browser. Because it's integrated with Observability, you can pull a specific failing trace directly into the Playground, tweak the prompt, and immediately see how the model responds. No code cycle.
How to use it
  • Import a versioned prompt from Prompt Management, or open a failing generation directly from a trace.
  • Set prompt variables to simulate different inputs without hardcoding them in the template.
  • Compare multiple variants side-by-side — different model, temperature, or system prompt.
  • Define tool schemas in JSON and test tool-calling behavior interactively.
  • Export the finalized prompt back to Prompt Management to version and deploy it.
Key Terms
prompt variables side-by-side comparison tool calling structured output

Evaluation · Scores
Attach structured evaluation results — numeric, boolean, or categorical — to any trace, span, or generation. The common currency of all evaluation methods.
The Problem
Without a scoring layer, LLM quality is invisible. You can't tell if a prompt change helped or hurt, catch regressions before shipping, or compare model variants systematically. Guesswork replaces data.
The Concept
A Score is a named, typed value on a trace or observation — from a human annotator, a code check, or an LLM judge. All three sources write to the same Score table, so you can directly compare automated vs. human judgments. Score Configs define the dimensions and value ranges used by all evaluators, keeping results consistent.
How to use it
  • Define Score Configs (name + type: numeric/boolean/categorical) once — reuse across all evaluators to keep dimensions consistent.
  • Write scores via SDK: client.score.create(trace_id=..., name="faithfulness", value=0.85)
  • Set up LLM-as-Judge evaluators in the UI to score live traces automatically on the same dimensions.
  • Query score trends in Score Analytics or pull raw data via API for CI/CD quality gates.
Key Terms
Score Score Config numeric / boolean / categorical Score Analytics
Evaluation · Evaluators — LLM-as-Judge
Use an LLM as an automated judge to score and reason about your application's outputs at scale — consistent, repeatable, human-like quality assessment.
The Problem
Manual evaluation doesn't scale past a few hundred samples. String-match or embedding-distance metrics miss nuanced dimensions like faithfulness, completeness, or tone. You need something that thinks about quality the way a human would — but runs on thousands of samples per minute.
The Concept
A judge LLM receives: input, output, optional reference, and a scoring rubric. It returns a score (numeric 0–1, categorical, or boolean) plus a reasoning trace. Evaluators run automatically on new observations in production, or on experiment datasets. Every judge run creates its own trace — fully debuggable.
How to set up
  • UI → Evaluators → "+ Set up Evaluator". Choose a Managed evaluator (ready-made, includes RAGAS) or write a Custom one with {{variable}} placeholders.
  • Set the judge model under LLM Connections — must support structured output (GPT-4o, Claude Sonnet, Gemini Pro all work).
  • Map trace fields to prompt variables with JSONPath. Use the live preview to confirm the prompt populates correctly.
  • Debug via Tracing filtered to environment: langfuse-llm-as-a-judge — every evaluation run is a trace with status Completed/Error/Delayed.
Key Terms
LLM-as-Judge scoring rubric observation evaluator RAGAS reasoning trace
Evaluation · Human Annotation
Collaboratively score traces, sessions, and observations through the Langfuse UI — building human baselines that automated evaluators can be benchmarked against.
The Problem
Automated metrics can't capture judgments that need domain expertise or business context. Teams need a structured, consistent way for multiple reviewers to label production data — and to create human-labeled ground truth to calibrate their LLM judges against.
The Concept
Annotators assign scores using predefined Score Configs (dimensions + value types). Annotation Queues let you work through large batches systematically. All human scores land in the same Score table as automated scores — directly comparable.
Annotation workflow
  • Step 1: Create Score Configs in your project (e.g. "correctness", "tone") — sets the evaluation dimensions and value types.
  • Step 2: Open any trace or observation detail view → click Annotate → select which Score Configs to apply.
  • Step 3: Set score values, optionally add a free-text comment documenting your reasoning.
  • Bonus: Use Annotation Queues to batch-route traces to reviewers for systematic coverage.
Key Terms
Score Config Annotation Queue human baseline ground truth
Evaluation · Datasets
A named, versioned collection of input/expected-output pairs — the stable test suite you run experiments against to detect regressions and compare changes.
The Problem
Without shared datasets, every developer runs ad-hoc tests — regressions go unnoticed, there's no baseline for comparing prompt or model changes, and you can't reproduce results from last week's experiment.
The Concept
A dataset item has an input and an optional expected_output. Items come from real production failures (linked via source_trace_id), synthetic examples, or manual entry. Datasets are versioned — you can pin a historical snapshot for reproducible re-runs even after items change.
How to use it
  • Create: langfuse.create_dataset("rag-failures") — optionally attach a JSON Schema to enforce item structure.
  • Add a production failure: create_dataset_item(input={...}, expected_output={...}, source_trace_id="...")
  • Or batch-add from the UI: Observations table → select rows → "Add to dataset" with field mapping.
  • Run an experiment: dataset.run_experiment(name="v2-test", task=my_llm_fn) — results stored as a named run, comparable side by side in the UI.
Key Terms
dataset item expected_output dataset version experiment run source_trace_id

Annotation QueueA batched list of traces routed to human reviewers for systematic scoring — ensures coverage without ad-hoc trace hunting.
async flushSDK mechanism that sends trace data in background threads after a request completes. Adds zero latency to your app's critical path.
AutomationA Monitor destination: Slack channel, HTTP webhook, or GitHub Actions dispatch. Fires on every Monitor severity state change.
compile()SDK method that substitutes {{variable}} placeholders in a prompt template with real values. Returns the final string (text) or message list (chat).
dataset itemOne row in a Langfuse Dataset: an input, optional expected_output, and an optional link to the source production trace via source_trace_id.
dataset versionA point-in-time snapshot of a dataset's items. Enables reproducible experiment re-runs even after items have been added or changed.
evaluation windowThe rolling time period a Monitor uses to aggregate its metric before comparing against thresholds (e.g. "last 1 hour" or "last 1 day").
experiment runOne execution of your task function against all items in a dataset. Stored as a named run in Langfuse for side-by-side comparison.
generationA span type that wraps a single LLM call. Auto-captures model name, prompt, completion, token counts, and inferred cost.
get_prompt()SDK method to fetch a prompt from Langfuse at runtime. Returns the "production"-labeled version by default; pin a specific version with version=N.
ground truthA human-verified expected output or manually annotated score. Used to calibrate automated evaluators and measure their agreement with human judgment.
human baselineA set of human-annotated scores across a representative trace sample. Used to benchmark how well automated evaluators agree with humans.
label: productionThe Prompt Management tag marking the live version of a prompt. The SDK's get_prompt() fetches this version by default at runtime.
LLM-as-JudgeUsing a capable LLM to evaluate another LLM's output against a scoring rubric. Returns a score and reasoning trace — human-like quality at scale.
