task dataset:seed · Bulk-create dataset items from this morning's traces scripts/seed_dataset.py
What it does
Fetches all traces tagged "sherlock-holmes" from Langfuse (up to 50), filters for ones with both input and output, and creates dataset items — each linked to its source trace via source_trace_id.
After running
Open Langfuse → Datasets → sherlock-eval. Items appear with the question, the model's answer, and a link back to the original trace. expected_output is blank — you fill that in for the items you know the right answer to.
# scripts/seed_dataset.py
lf = Langfuse(public_key=..., secret_key=...)

# 1. Pull recent traces with question + answer
traces = lf.get_traces(
    tags=["sherlock-holmes"],
    limit=50,
)
candidates = [t for t in traces.data if t.input and t.output]

# 2. Create dataset if it doesn't exist (idempotent)
lf.create_dataset(name="sherlock-eval")

# 3. Add one item per trace — linked to source
for trace in candidates:
    lf.create_dataset_item(
        dataset_name="sherlock-eval",
        input={"question": trace.input["question"]},
        expected_output=None,    ← you fill this in
        source_trace_id=trace.id,  ← links to live trace
    )
    print(f"  + {trace.input['question'][:70]}")
Before scoring — define your dimensions once
Score Configs: UI → Settings → Score Configs
faithfulness — numeric 0.0–1.0. Does the answer contain only information supported by the retrieved context? Used by LLM-as-Judge and RAGAS.
completeness — numeric 0.0–1.0. Does the answer address the full question, or does it dodge parts of it? Useful for multi-part questions.
user_feedback — numeric 0 or 1, written by the 👍/👎 buttons in the frontend. Already flowing into Langfuse — this config makes it visible in Score Analytics.
retrieval_quality — categorical good / partial / poor. Annotated manually when reviewing dataset items — captures retrieval failures that faithfulness alone misses.
Define these now — every evaluator in Block B will write to the same Score Config names, making all sources directly comparable in Score Analytics.

task experiment:run · Run dataset through live pipeline, store named run scripts/run_experiment.py
What it does
Pulls each dataset item, runs the question through the live RAG pipeline, stores the result as a named experiment run in Langfuse, and scores faithfulness synchronously.
item.observe()
This context manager creates a trace linked to the dataset item and the run name. All child spans hang off it. The Langfuse UI shows runs side by side under the dataset.
Two scores per item
The script scores synchronously for immediate feedback. Langfuse's managed evaluator also scores asynchronously using the full retrieval span context.
dataset = lf.get_dataset("sherlock-eval")

for item in dataset.items:
    question = item.input["question"]

    # observe() links trace to dataset item + run
    with item.observe(run_name=run_name) as trace:
        trace.update(tags=[settings.llm_model, "experiment"])

        answer_parts = []
        async for token in stream_answer(question, repo, trace, settings):
            answer_parts.append(token)

        answer = "".join(answer_parts)
        trace.update(output={"answer": answer})

    # synchronous faithfulness judge
    faith = await judge_faithfulness(question, context, answer, model)
    lf.score(trace_id=trace.id, name="faithfulness", value=faith)
    scores.append(faith)
    print(f"  {'✓' if faith else '✗'} {question[:65]}")
Model comparison · Change one variable, run twice, compare in Langfuse Taskfile.yaml · UI → Experiments
The principle
A valid experiment changes exactly one variable — the model — against a fixed dataset with a fixed score. Everything else stays constant.
In Langfuse UI
Datasets → sherlock-eval → Experiments tab. Both runs appear side by side. Click any item to see the per-question score delta: did the cheaper model hold up?
What to look for
If gpt-4o-mini is within 5% faithfulness of gpt-4o, the cost saving is worth it. If it drops 20%+, it isn't.

task regression:test · Quality gate — exits non-zero if faithfulness < 0.70 scripts/run_experiment.py --threshold 0.70
Same script, one flag
task regression:test calls run_experiment.py --threshold 0.70. All the same scoring logic; if average faithfulness falls below the threshold, the process exits 1 — blocking CI.
Wire to GitHub Actions
Add task regression:test as a step in your CI workflow on the main branch. A prompt change that tanks faithfulness gets caught before it ships.
Take-home
RAGAS full suite (context recall, precision, answer relevancy) is available as Langfuse managed evaluators. Monitors (UI → Monitors) can alert on Slack when faithfulness drops in production.
# Taskfile.yaml
"regression:test":
  desc: Run experiment, exit non-zero if avg faithfulness < 0.70
  cmds:
    - uv run python scripts/run_experiment.py --threshold 0.70


# scripts/run_experiment.py (end of run())
avg = sum(scores) / len(scores)
print(f"Avg faithfulness: {avg:.2%} (threshold: {threshold:.2%})")

if threshold is not None and avg < threshold:
    print(f"FAIL — {avg:.2f} below {threshold:.2f}")
    sys.exit(1)  ← CI sees non-zero → blocks merge

Additional ideas — now you have the evaluation loop to measure them
Three ways to improve retrieval quality
HyDE — embed a fake answer Ask the LLM to write a short hypothetical answer first, then embed that instead of the raw question. A fake answer uses the vocabulary of the source text and lands much closer to the real passage in embedding space.
Query rewriting — match the corpus Ask the LLM to rephrase the question as keywords or a declarative statement that matches the language of the source text. Cheaper than HyDE — good first A/B candidate.
Multi-query retrieval — cast a wider net Generate 2–3 alternative phrasings, run each through the vector search, deduplicate and merge the results. Multiple retrieval spans visible in Langfuse per turn.
Your dataset + regression gate is already in place — run any of these ideas through task regression:test to measure whether they actually help.
